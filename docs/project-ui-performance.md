# Project UI Performance

Вкладка "Единое здание" является техническим preview для проверки, что один `ProjectInput`
транслируется во все расчетные блоки.

## Почему было медленно

До оптимизации каждое изменение поля формы сразу запускало:

- `calculateProjectWithSummary()`;
- `buildBuildingSpecification()`.

Обе функции считают все шесть блоков здания, включая тяжелые VELICAN Excel/workbook oracle
модули для подкрановой балки, оконного ригеля и балки покрытия. Поэтому ввод текста или изменение
числового поля могло заметно тормозить UI.

## Текущее поведение

В `ProjectApp` теперь есть два состояния:

- `draftProjectInput` - редактируется формой и не запускает тяжелый расчет;
- `calculatedProjectInput` - используется для summary и specification.

Расчет выполняется:

- один раз при первом открытии вкладки на `defaultProjectInput`;
- при нажатии кнопки "Рассчитать";
- при нажатии "Сбросить", когда оба состояния возвращаются к `defaultProjectInput`.

Если форма отличается от последнего рассчитанного ввода, UI показывает заметку:

`Есть несохраненные изменения. Нажмите «Рассчитать», чтобы обновить результаты.`

## Memoization

`calculateProjectWithSummary(calculatedProjectInput)` и
`buildBuildingSpecification(calculatedProjectInput)` находятся внутри `useMemo`.

Изменение полей формы меняет только draft-состояние, поэтому тяжелые вычисления не запускаются
на каждый `onChange`.

## Время расчета

В summary выводится строка "Расчет выполнен за ... мс". Это технический ориентир для дальнейшей
оптимизации, а не инженерный расчетный параметр.

## Что дальше

Oracle-блоки остаются тяжелыми. Следующие возможные этапы оптимизации:

- cache для одинаковых расчетных inputs;
- разделение native и oracle расчетов по режимам;
- Web Worker для тяжелых workbook/oracle блоков;
- постепенный native rewrite oracle-блоков после инженерной приемки.

Формулы, mapper-логика, acceptance tests и старые вкладки этим изменением не затрагиваются.
